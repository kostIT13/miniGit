## Что такое miniGit?

**miniGit** — это учебная реализация системы контроля версий, написанная на чистом C.


### Цели проекта

1. **Понимание персистентности** — любая предыдущая версия остаётся неизменной
2. **Структурное разделение** — неизменённые файлы не копируются, а используются по ссылке
3. **Хэширование содержимого** — адресация объектов по хэшу, а не по имени
4. **Цепочка коммитов** — каждый коммит ссылается на родителя, образуя историю

##  Ключевые концепции

### Персистентность (Неизменяемость)

```c
Commit *c1 = add_file(head, "file.txt", "v1");
Commit *c2 = add_file(c1, "file.txt", "v2");

// c1 и c2 — разные объекты в памяти
// При изменении c2, c1 остаётся неизменным
assert(strcmp(get_file_content(c1, "file.txt"), "v1") == 0);  // correct
assert(strcmp(get_file_content(c2, "file.txt"), "v2") == 0);  // correct
```

### Структурное разделение (Structural Sharing)
```bash
Коммит 1: [fileA@hash1] [fileB@hash2]
                    │
Коммит 2: [fileA@hash1] [fileB@hash3]  ← fileA НЕ копируется!
                    │
Коммит 3: [fileA@hash1] [fileC@hash4]  ← fileA всё ещё общий!
```

* Неизменённые файлы хранятся один раз в пуле блобов
* Разные коммиты ссылаются на одни и те же хэши
* Экономия памяти: при 100 коммитах с 10 файлами, где меняется 1 файл — хранится ~110 блобов, а не 1000

### Хэширование содержимого
```bash
// Хэш вычисляется от содержимого, а не от имени файла
char hash[HASH_HEX_LEN + 1];
content_hash("Hello World", 11, hash);  // → "2aae6c35c94fcfb415dbe95f408b9ce91ee846ed"

// Одинаковое содержимое = одинаковый хэш = один объект
blob_save("test", 4, hash1);  // hash1 = "9845fde1b08564a7c8c4cb6d0158bb3b19982201"
blob_save("test", 4, hash2);  // hash2 = "9845fde1b08564a7c8c4cb6d0158bb3b19982201" (тот же!)
```

### Цепочка коммитов

```bash
commit abc123... (HEAD -> master)
Author: miniGit
Date:   2024-04-24

    Add feature X

commit def456...
Author: miniGit  
Date:   2024-04-23

    Initial commit
```

## Фишки и особенности

miniGit не просто учебный проект — это полноценная система контроля версий с рядом интересных возможностей:

| Фишка | Описание | Пример |
|-------|----------|--------|
| **Мгновенное создание версий** | Каждое изменение создаёт новый коммит за O(1) от количества изменённых файлов | `add_file(commit, "file.txt", "new")` → новый коммит |
| **Автоматическая дедупликация** | Одинаковое содержимое файлов хранится один раз независимо от имени или пути | Два файла с текстом "Hello" → один blob |
| **Иммутабельные структуры** | Все коммиты неизменяемы — можно безопасно передавать между потоками | `commit1 != commit2`, но общие поддеревья |
| **Минимальный footprint** | Хранит только разницу между версиями, а не полные копии | 1000 коммитов с 1 изменением в каждом → ~1001 blob |
| **Человекочитаемые хэши** | 40-символьные хэши (улучшенный DJB2), визуально похожие на SHA-1 | `2aae6c35c94fcfb415dbe95f408b9ce91ee846ed` |
| **Полная история в памяти** | Вся цепочка коммитов доступна для навигации без чтения с диска | `print_history(head)` показывает всю историю |
| **Кроссплатформенность** | Работает на Windows, Linux, macOS без зависимостей | `make` на любой ОС |

## Сравнение с Git

| Возможность | Git | miniGit |
|-------------|-----|---------|
| Хранилище объектов | `.git/objects/` (сжатые) | `.minigit/objects/` (plain text) |
| Хэш-алгоритм | SHA-1 (20 байт) | Улучшенный DJB2 (20 байт) |
| Структура коммита | tree, parent, author, message | tree, parent, message, hash |
| Ветвление | Да (refs/heads/) | Нет (только линейная история) |
| Слияние | Да (merge) | Нет |
| Индекс (staging) | Да (git add) | Нет (изменения сразу коммитятся) |
| CLI команд | 150+ | 10 основных |
| Размер кода | ~2 млн строк | ~2000 строк |

> **Примечание**: miniGit — это упрощённая реализация для обучения, а не замена Git.

## Визуализация работы

```
Коммит A (abc123)
    │
    ├── file1.txt → blob_hash1
    ├── file2.txt → blob_hash2
    └── subdir/
        └── file3.txt → blob_hash3
          │
Коммит B (def456) ← изменён только file2.txt
    │
    ├── file1.txt → blob_hash1  (общий с A)
    ├── file2.txt → blob_hash4  (новый blob)
    └── subdir/
        └── file3.txt → blob_hash3  (общий с A)
```

**Ключевое наблюдение**: Неизменённые файлы (file1.txt, file3.txt) используют те же blob-объекты, что и в предыдущем коммите — это и есть структурное разделение.

## Практические примеры

### Создание простого репозитория с историей
```c
#include "minigit.h"

int main() {
    // Инициализация
    Commit *c0 = init_repo();
    
    // Добавляем файлы
    Commit *c1 = add_file(c0, "README.md", "# MiniGit Project");
    Commit *c2 = add_file(c1, "src/main.c", "int main() { return 0; }");
    Commit *c3 = add_file(c2, "README.md", "# MiniGit Project\n\nVersion 2.0");
    
    // Удаляем файл
    Commit *c4 = remove_file(c3, "src/main.c");
    
    // Фиксируем коммиты
    Commit *final1 = commit(c1, "Add README");
    Commit *final2 = commit(c2, "Add source code");
    Commit *final3 = commit(c3, "Update README");
    Commit *final4 = commit(c4, "Remove main.c");
    
    // Просмотр истории
    print_history(final4);
    
    return 0;
}
```

## Советы по расширению

Хотите добавить новые функции в miniGit? Вот несколько идей:

1. **Ветвление (branches)**
   - Добавить структуру `Branch` с указателем на коммит
   - Реализовать `create_branch()`, `switch_branch()`
   - Хранить ветки в `.minigit/refs/heads/`

2. **Слияние (merge)**
   - Алгоритм трёхстороннего слияния для деревьев
   - Обнаружение конфликтов и их разрешение

3. **Удалённые репозитории**
   - Протокол передачи объектов по HTTP/SSH
   - Команды `push`, `pull`, `fetch`

4. **Графический интерфейс**
   - Использовать SDL или GTK для визуализации истории
   - Показывать дерево файлов с цветовой индикацией изменений

5. **Производительность**
   - Кэширование часто используемых blob-объектов
   - Сжатие объектов (zlib)
   - Индексирование для быстрого поиска по хэшам

6. **Дополнительные команды CLI**
   - `minigit diff <commit1> <commit2>` — сравнение версий
   - `minigit log --graph` — графическое представление истории
   - `minigit checkout <hash>` — восстановление состояния

## Ограничения (и как их обойти)

| Ограничение | Причина | Обходной путь |
|-------------|---------|---------------|
| Линейная история | Нет ветвления | Реализовать ветвление (см. выше) |
| Нет сжатия | Упрощение кода | Добавить zlib-сжатие в blob_save |
| Фиксированный размер children[10] | Статический массив | Заменить на динамический массив/список |
| Нет работы с бинарными файлами | Текстовый режим | Использовать `fread`/`fwrite` с двоичным режимом |
| Нет транзакционности | Простота | Добавить журналирование (WAL) |

## Сборка и запуск

### Требования

* Компилятор C99+: gcc, clang, MSVC
* Утилита make (или ручная компиляция)
* Windows: MinGW или Visual Studio с поддержкой C99

### Сборка
```bash
# Linux/macOS
make clean
make

# Windows (MinGW)
mingw32-make clean
mingw32-make

# Windows (MSVC)
nmake -f Makefile.msvc
```

### Базовое использование (через CLI)
```bash
# Инициализация репозитория
./minigit.exe init_repo

# Добавление файла
./minigit.exe add_file readme.txt "Hello MiniGit!"

# Фиксация коммита
./minigit.exe commit "Initial commit"

# Просмотр истории
./minigit.exe print_history

# Чтение файла из текущей версии
./minigit.exe get_file_content readme.txt

# Управление ветками
./minigit.exe create_branch feature abc123def456...
./minigit.exe get_branch_head feature
./minigit.exe checkout abc123def456...
```

# API Справочник miniGit

## Основные функции

| Функция | Описание | Параметры | Возвращает |
|---------|----------|-----------|------------|
| `Commit* init_repo(void)` | Создаёт пустой репозиторий и возвращает начальный коммит | — | `Commit*` — начальный коммит или `NULL` |
| `Commit* add_file(Commit *old_commit, const char *path, const char *content)` | Добавляет/обновляет файл в новой версии | `old_commit` — текущий коммит<br>`path` — путь к файлу<br>`content` — содержимое | `Commit*` — новый коммит (старый неизменен) |
| `Commit* remove_file(Commit *old_commit, const char *path)` | Удаляет файл из новой версии | `old_commit` — текущий коммит<br>`path` — путь к файлу | `Commit*` — новый коммит или тот же, если файла не было |
| `Commit* commit(Commit *state, const char *message)` | Фиксирует состояние с сообщением | `state` — промежуточное состояние<br>`message` — сообщение коммита | `Commit*` — финальный коммит с хэшем |
| `char* get_file_content(Commit *commit, const char *path, size_t *out_len)` | Получает содержимое файла из версии | `commit` — коммит<br>`path` — путь к файлу<br>`out_len` — длина контента (опционально) | `char*` — строка с контентом или `NULL` |
| `bool get_file_exists(Commit *commit, const char *path)` | Проверяет наличие файла в версии | `commit` — коммит<br>`path` — путь к файлу | `bool` — `true`/`false` |
| `void print_commit(Commit *commit)` | Выводит информацию об одном коммите | `commit` — коммит | — |
| `void print_history(Commit *commit)` | Выводит цепочку коммитов назад до начального | `commit` — любой коммит | — |
| `void print_files(Commit *commit)` | Выводит список всех файлов в версии | `commit` — коммит | — |
| `MinigitStatus create_branch(const char *branch_name, Commit *commit)` | Создаёт новую ветку, указывающую на коммит | `branch_name` — имя ветки<br>`commit` — целевой коммит | `MINIGIT_OK` при успехе, иначе код ошибки |
| `Commit* get_branch_head(const char *branch_name)` | Возвращает головной коммит ветки | `branch_name` — имя ветки | `Commit*` — коммит или `NULL` |
| `MinigitStatus checkout(Commit *commit)` | Переключает HEAD на указанный коммит (detached HEAD) | `commit` — коммит для переключения | `MINIGIT_OK` при успехе, иначе код ошибки |
